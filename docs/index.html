Why I built this

Most RAG demos stop at "embed a PDF, ask a question." That is a retrieval lookup, not an agent. I wanted to build something closer to what production systems actually need: a LangGraph ReAct agent that reasons about which tool to use, streams its trace back to the UI in real time, maintains conversation history across turns, and handles multiple documents simultaneously.

The retrieval side was the other gap I wanted to close. Vanilla cosine similarity over chunks is fragile. Short or vague queries return noise. So I built a multi-stage pipeline: HyDE query rewriting to generate a hypothetical answer before retrieval, hybrid BM25 + dense search for breadth, and a cross-encoder to re-rank the survivors. The difference in recall is significant.

Deployment was a deliberate constraint: one Docker container, no external services required. nginx + supervisord orchestrates the Next.js frontend and FastAPI backend inside a single image, with in-memory Qdrant as fallback. It runs identically locally and on HuggingFace Spaces.

How it works

End-to-end request flow, from browser to answer:

Browser (Next.js 15) | type a question, hit Send v FastAPI /chat +-- Input guardrails (prompt injection patterns, length limits) +-- Session memory lookup (20-turn history per UUID) | v LangGraph ReAct Agent decides which tool to call | +-- rag_search | 1. HyDE rewrite LLM generates a hypothetical answer, use that for embedding lookup | 2. Hybrid retrieve BM25 + dense via EnsembleRetriever (k=12 each) | 3. Re-rank cross-encoder/ms-marco-MiniLM-L-6-v2, top 6 chunks | 4. Return chunks with [Chunk N | doc_name | page | strategy] citations | +-- web_search Tavily API, 4 results when document does not contain the answer | v LLM (OpenAI gpt-4o-mini / gpt-4o or HuggingFace Mistral / Zephyr / Llama-3) | streams tokens via SSE v Browser renders token stream + collapsible agent trace + source citations

Key features

01

HyDE Query Rewriting

Before retrieval, the LLM generates a short hypothetical passage that would answer the question. The embedding of that passage is used for vector lookup instead of the raw query. Recall on short or ambiguous questions improves substantially. The original query is still used for re-ranking.

02

Hybrid BM25 + Dense Retrieval

BM25 and dense vector search run in parallel and merge via EnsembleRetriever (weights 0.4 / 0.6). BM25 catches exact keyword matches that dense search misses; dense search handles semantic paraphrasing. Together they consistently outperform either alone.

03

Cross-Encoder Re-ranking

The top-12 candidates from hybrid retrieval are scored by ms-marco-MiniLM-L-6-v2, a cross-encoder that reads query and passage together rather than comparing embeddings in isolation. Top 6 are passed to the LLM, cutting noise and token cost.

04

Multilingual Embeddings

Embeddings use paraphrase-multilingual-MiniLM-L12-v2 with 50+ language support and the same 384-dimensional space as the English-only model. Documents in non-English languages are indexed and retrieved correctly with no code changes.

05

LangGraph ReAct Agent

The agent loop uses LangGraph's MessagesState and ToolNode. It decides whether to call rag_search or web_search based on the question, can chain multiple tool calls, and streams all intermediate steps back to the UI via SSE.

06

Input and Output Guardrails

Input is checked against 9 regex patterns covering prompt injection and jailbreak attempts before the agent sees it. Oversized queries (>2000 chars) are rejected with a clear error. Output is truncated at 8000 chars and flagged if hallucination signal phrases are detected.

07

RAG Evaluation Metrics

After every answer, the user can trigger an evaluation pass that computes three embedding-based scores: answer relevancy, faithfulness to retrieved context, and context recall, entirely via cosine similarity. No external eval API required.

08

Single-Container Deployment

A multi-stage Dockerfile builds the Next.js frontend and Python backend into one image. nginx routes /chat, /ingest, and other API paths to FastAPI on port 8000; all other traffic goes to Next.js on port 3000. supervisord manages both processes. Qdrant falls back to in-memory if no external URL is set.
